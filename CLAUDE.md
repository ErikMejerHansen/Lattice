# Lattice

A personal physics tutor. Single user (Erik). Lessons are pre-generated by a scheduled Claude Code routine reading from a wishlist; the React app reads static JSON from the repo and renders them. No backend, no API key in the browser, deployed via GitHub Pages.

## What this file is

Guidance for Claude Code (and the human) when **building** Lattice. The routine that **generates lessons** has its own separate CLAUDE.md inside `/routine/CLAUDE.md` once we get there. Don't conflate the two.

## Build philosophy

Vertical slices, ten of them. Each slice ends with something runnable that's worth shipping if we stopped there. We do not work ahead. We do not build infrastructure for slice 6 during slice 2. If a clever idea occurs mid-slice, it goes in `wishlist.md` and we keep moving.

The current slice is tracked at the bottom of this file under "Status." Update it after each slice ships.

## The slices

1. **One static lesson rendered.** Hand-written JSON, bare React, `npm run dev`, read it in a browser.
2. **GitHub Pages deploy.** Same lesson, now live, viewable on phone.
3. **Claude Code generates one lesson manually.** Terminal-driven, committed by hand.
4. **App lists all lessons.** Index manifest, tappable list.
5. **Validation script.** `validate-lesson.mjs` runs cleanly against existing lessons.
6. **Routine spec, manual trigger.** Claude Code executes the routine end-to-end under supervision.
7. **Viz iframe rendering.** Embed bespoke HTML viz files from lesson JSON.
8. **Schedule the routine.** Daily 7am run.
9. **Constellation view.** Replace list with concept graph.
10. **Notes loop.** Per-section note input, queue review at lesson end, copy-paste to wishlist.

## What's true about Lattice (decisions already made)

These are settled. Don't relitigate.

- **Stack:** React + Vite, deployed to GitHub Pages. Build output goes to `/docs`. Pages serves from `/docs` on `main`.
- **No backend.** No proxy, no Worker. Lessons are static files in the repo.
- **No API key in the browser.** All Claude calls happen at routine-generation time, not runtime.
- **Lesson schema:** structured JSON with blocks (`p`, `math`, `viz`). One viz per lesson maximum, embedded as a sandboxed iframe pointing at a sibling HTML file.
- **Aesthetic:** Hamming's *The Art of Doing Science and Engineering*. Cream paper (#f4eee2), rust accent (#9c3a22), Source Serif 4, IBM Plex Mono for category labels. Minimal chrome, type-led hierarchy, no rounded-everything.
- **Voice:** direct, story-led, historical Connections-style asides. The "BBC Connections feel" is the part the user loves most. Lean into it.
- **Punctuation in generated content:** no em-dashes (use commas, colons, parentheses). Plain ASCII apostrophes. Unicode for math symbols in prose.
- **Constellation states:** `mastered`, `in_progress`, `available` (lesson exists, not started), `queued` (user-flagged for future), `suggested` (only used in the wishlist, not in the graph).
- **Naming:** the project is "Lattice." Components, files, variables follow ordinary React conventions; no themed naming.

## Working agreements

**Ask before assuming.** When a slice begins, read this file plus the relevant prior slice's output. If a decision feels load-bearing and isn't covered here, ask the human; don't pick.

**Stop and log when flagging.** When something blocks progress (missing infrastructure, ambiguous spec, conflicting instructions), default to: append a one-line entry to `state/routine_log.md` (or the appropriate log for the context), commit it, and stop. Don't improvise around the blocker. Don't half-build infrastructure to keep moving.

**One slice at a time.** If a request seems to span multiple slices, name the split and propose the smallest first slice. Don't quietly do extra work.

**No premature abstraction.** Inline the lesson rendering for slice 1. Extract a `Block` component when slice 4 makes us render multiple lessons. Don't build a plugin system for viz types in week one.

**Commit messages:** present tense, lowercase prefix. `slice 1: render one static lesson` is fine. `feat: add lesson rendering capability` is not.

**Branches:** main only, until we have a reason for branches. This is a one-person project.

**Tests:** none for the React app. Schema validation lives in `scripts/validate-lesson.mjs` (slice 5+) and is the only enforced quality gate. Visual correctness is checked by looking at it.

**Dependencies:** Vite, React, react-router (only when slice 4 needs it), KaTeX (only when math blocks exist). No state management library — `useState` is fine for an app this size. No CSS framework — inline styles or a single `app.css` file.

## Open questions

These will get resolved as we go. Listed here so they're visible.

- **Routing.** Do we need URLs per lesson (`/lattice/lessons/carnot-cycles`) or is in-app state enough? Decide at slice 4.
- **Viz CDN policy.** The routine spec says cdnjs only. The app shell may also need to load KaTeX for math blocks — same policy, different consumer. Confirm at slice 1 or 2.
- **PWA / home screen.** Iconography, manifest, "Add to Home Screen" flow. Not blocking anything; pick up around slice 8 when daily use becomes real.
- **Storage.** Mastered/in_progress state has to live somewhere. localStorage is fine for now (single device, single user). Revisit only if we hit iOS storage eviction in practice.
